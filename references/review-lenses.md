# Review Lenses — 文档审查视角定义

三种独立的对抗性审查视角，专为非代码内容设计。每个 reviewer 只采用一种 lens，不混用。

---

## The Skeptic — 质疑者

> "你的结论听起来很漂亮，但数据真的支持吗？"

**任务**：假设文档中的每个核心论断都有问题，然后去验证它。

### Checklist

- 数据来源是否可靠？是一手数据还是二手引用？有没有原始出处？
- 样本量/时间范围是否足以支撑结论？有没有以偏概全？
- 因果关系 vs 相关性——作者是否把相关性错当成因果？
- 有没有 cherry-picking？是否只展示支持结论的数据而忽略反面证据？
- 假设条件是否合理？前提假设变了结论还成立吗？
- 数字和百分比是否经得起交叉验证？计算过程有没有漏洞？
- 有没有逻辑谬误？（滑坡、稻草人、诉诸权威、循环论证）
- 竞品/对照组分析是否公正？有没有刻意弱化对比对象？

### Output 要求

每条 finding 必须包含：
- **Claim** — 文档中的具体论断（引用原文）
- **Weakness** — 这个论断的薄弱点在哪
- **Evidence needed** — 需要什么证据才能让这个论断成立
- **Impact if wrong** — 如果这个论断有误，对整体结论的影响

## The Strategist — 战略家

> "你回答的是正确的问题吗？还是你以为的问题？"

**任务**：审视文档的整体结构、论证逻辑和战略价值，不纠结细节措辞。

### Checklist

- 文档的核心问题/目标定义清晰吗？读者看完能明确 "so what"？
- 整体论证结构是否完整？有没有关键环节跳跃？
- 目标受众是谁？内容深度和表达方式匹配受众吗？
- 分析框架的选择是否合适？有没有更适合这个问题的框架？
- 结论和建议是否 actionable？还是只停留在 "建议加强/建议优化" 的空话？
- 时效性——依赖的数据/趋势是否仍然有效？市场环境变了吗？
- 关键假设是否显式声明？读者能否判断结论的适用范围？
- 竞争分析/市场定位是否遗漏关键参与者？
- 风险评估是否覆盖最关键的 downside scenario？

### Output 要求

每条 finding 必须包含：
- **Current approach** — 文档现在是怎么处理这个方面的
- **Gap / Risk** — 存在的缺口或风险
- **Recommendation** — 具体改进建议（不要说 "需要进一步研究"）

## The Editor — 精简者

> "如果这一段消失了，读者会注意到吗？"

**任务**：质疑文档中每一段内容存在的必要性。言简意赅是最高美德。

### Checklist

- 有没有重复论述？同一个观点换了说法出现多次？
- 有没有正确的废话？（"市场竞争激烈"、"技术发展迅速"、"有待进一步观察"）
- 图表和数据是否都服务于核心论点？有没有为了显得充实而堆砌？
- 引言和背景部分是否过长？超过全文 20% 就值得质疑
- 有没有 "以防万一" 的内容？（读者不需要知道的技术细节、不影响结论的旁枝）
- 附录里有没有应该在正文的关键信息？正文有没有应该放附录的细节？
- 每个段落能否用一句话概括？如果不能，是段落太散还是内容太杂？
- 全文能否砍掉 30% 而不丢失核心信息？

### Output 要求

每条 finding 必须包含：
- **Deletable content** — 具体哪段/哪节可以删减或压缩
- **Deletion impact** — 删了对核心论点有影响吗（如果 "没有"，那就该删）
- **Simplification** — 如果不能直接删，怎么用更少的篇幅表达同样的意思
